# Architecture

이 문서는 노후 수입 설계 서비스의 초기 아키텍처 방향을 정의한다. 현재 단계의 목표는 신뢰 가능한 월별 현금흐름 프로젝션을 만들기 위한 도메인 경계, 데이터 흐름, 검증 지점을 명확히 하는 것이다.

## System Context

사용자는 본인과 현재 배우자의 생년월일, 현재 근로소득, 퇴직 예정 또는 희망퇴직 가능성, 보유 자산, 임대 수입, 연금 및 금융자산을 입력한다. 시스템은 한국 기준 세후 현금흐름으로 현재부터 본인이 100세가 되는 해까지 월 단위 수입을 계산하고, 연령별 현금흐름과 목표 월수입 달성 가능성을 보여준다.

```text
User
  -> Onboarding and asset intake
  -> Scenario builder
  -> Projection engine
  -> Reliability harness
  -> Monthly income report
```

## Core Components

### 1. Product Intake

- 본인 및 배우자 생년월일
- 한국 거주 및 원화 기준 고정
- 현재 재직 여부, 근로소득, 은퇴 목표 연령
- 희망퇴직 시점, 퇴직금, 희망퇴직위로금
- 본인이 100세가 되는 해까지의 분석 기간
- 부동산, 금융자산, 연금, 기타 현금흐름
- 국민연금, 퇴직연금, 개인연금, IRP, 주택연금 등 연금별 수급 개시 시점
- 임대소득세, 재산세, 종합소득세, 물가상승률 등 가정값
- 퇴직 이후 목표 월수입

### 2. Domain Model

초기 도메인 객체는 다음을 중심으로 설계한다.

- `Household`: 사용자와 선택 입력인 현재 배우자, 한국 기준 분석 기간
- `Person`: 생년월일, 재직 상태, 은퇴 연령, 연금 개시 연령
- `EmploymentIncome`: 근로소득, 퇴직금, 희망퇴직위로금, 퇴직 이벤트
- `Asset`: 부동산 수입/비용, 예금, 투자금, IRP, 연금, 기타 수입원
- `IncomeStream`: 월별 또는 연별 수입 규칙
- `ExpenseAdjustment`: 세금, 유지보수비, 관리비, 대출 상환 등 비용
- `RuleSource`: 국민연금, 주택연금, 세금 등 외부 규칙의 출처와 버전
- `Recommendation`: 목표 월수입 달성을 위한 인출 전략 또는 추가 저축액
- `Scenario`: 기본, 보수, 낙관 등 가정 묶음
- `Projection`: 월별 수입 산출 결과

### 3. Projection Engine

프로젝션 엔진은 입력값과 시나리오를 받아 월별 수입 타임라인을 생성한다.

- 생년월일 기반 연령 계산
- 은퇴 전 근로소득과 은퇴 후 소득의 연결
- 퇴직금과 희망퇴직위로금의 일시 유입 및 이후 운용 반영
- 국민연금, 퇴직연금, 개인연금, IRP, 주택연금의 개시 시점별 월수입 반영
- 공식 또는 검증된 규칙 출처를 기반으로 한 연금 수령액 추정
- 부동산 월 수입과 비용, 수입원별 상승률 반영
- 투자자산 인출률 또는 배당/이자 수익 반영
- 현재 주식 투자액의 예상수익률 반영
- 목표 월수입 대비 부족액을 맞추기 위한 추천 인출 전략 또는 추가 저축액 산출

초기 범위에서 제외하는 계산은 배우자 사망, 상속, 유족연금, 부동산 매각, 공실률이다.

### 4. Reliability Harness

Harness Engineering은 계산 로직의 신뢰성을 확보하는 별도 레이어로 둔다.

- 결정적 테스트 케이스: 고정 입력에 대한 월별 결과 스냅샷
- 속성 기반 테스트: 월별 기간, 나이 계산, 수입 합계 불변식 검증
- 경계값 테스트: 윤년 생일, 월말 생일, 배우자 없음, 음수 현금흐름
- 회귀 테스트: 세법/연금 규칙 변경 전후 결과 비교
- 설명 가능성 테스트: 산출 결과의 근거 항목 추적

### 5. Reporting Layer

- 월별 총수입
- 수입원별 breakdown
- 사용자와 배우자 연령
- 세후 현금흐름 기준 결과
- 명목/실질 금액 선택 표시
- 목표 월수입 대비 부족 또는 초과
- 추천: 현재 자산의 월별 수령 방식, 은퇴 전 추가 저축 필요액, 원금 소진 예상 시점
- 리스크 플래그: 현금흐름 공백, 과도한 단일 자산 의존, 물가 조정 부족, 공식 규칙 미검증

## Data Flow

```text
Input Form
  -> Validation
  -> Normalized Domain Model
  -> Scenario Assumptions
  -> Projection Engine
  -> Harness Checks
  -> Report API
  -> UI
```

## Reliability Boundaries

계산 엔진은 UI나 저장소 구현과 분리한다. 동일한 입력과 동일한 가정값 및 동일한 규칙 출처 버전은 항상 동일한 프로젝션을 반환해야 하며, 모든 금액/날짜 계산은 원화와 월 단위 타입을 명시한다.

## Initial Technology Assumptions

아직 구현 언어와 프레임워크는 결정하지 않았다. 다만 초기 선택 기준은 다음과 같다.

- 날짜 및 금융 계산 테스트가 쉬울 것
- 문서화된 도메인 규칙을 코드로 옮기기 쉬울 것
- API와 계산 엔진을 분리할 수 있을 것
- CI에서 빠르게 하네스 테스트를 실행할 수 있을 것
